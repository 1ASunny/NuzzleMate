### 1.5 权威依据与合规边界

本 PRD 的医疗安全边界按“AI 辅助分诊，兽医专业承接”设计。根据《动物诊疗机构管理办法》，动物诊疗机构可以通过本机构备案从业的执业兽医师，利用互联网等信息技术开展动物诊疗活动，但活动范围不得超出动物诊疗许可证核定范围。因此，平台不能让 AI 独立完成诊断、治疗、配药、给药等诊疗行为，涉及诊断、治疗、处方等事项必须由合规动物诊疗机构和执业兽医承接。

兽用处方药必须凭兽医处方笺购买和使用，兽用非处方药才可不凭处方自行购买并按说明书使用。因此，系统允许“用药提醒、处方记录、药品说明解释”，但不允许 AI 直接开具处方药、推荐处方药剂量或引导无处方购药。

### 2.3 业务主流程

```text
用户登录
↓

↓

↓

```

#### AI 问诊流程

```text
用户点击 AI 问诊
↓
选择宠物
↓
输入文字 / 上传图片 / 语音描述
↓
系统读取宠物档案
↓
AI 判断症状类别
↓
多轮追问关键信息
↓
急症红旗规则判断
↓
风险分级
↓
生成病情摘要
↓
输出下一步建议
│
├── 低风险：观察 + 创建健康记录
├── 中风险：建议线上兽医问诊
├── 高风险：建议尽快线下医院就医
```

```text

↓

医生回复建议
```

### 3.3 通用规则

#### AI 医疗安全规则

| 编号 | 规则 |
|---|---|
| AI-R001 | AI 结果页必须展示“AI 辅助分诊，不能替代执业兽医诊断” |

| AI-R007 | 高风险病例进入后台风险病例池，支持人工抽检 |
| AI-R008 | AI 回答需使用结构化模板：风险等级、可能方向、建议行动、就医条件、下一步按钮 |

## 4. 功能详细需求

### F-001 登录 / 注册

**功能目的：** 让用户进入系统并建立唯一身份，支持宠主、医生、运营人员进入对应端。

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| mobile | string | 是 | 手机号 |
| sms_code | string | 是 | 验证码 |
| role | enum | 是 | pet_owner / doctor / admin |
| password | string | 否 | 后台账号密码登录使用 |
| agreement_checked | boolean | 是 | 是否勾选协议 |

操作流程：

```text
进入登录页
↓
输入手机号
↓
点击获取验证码
↓
输入验证码
↓
勾选协议
↓
点击登录 / 注册
↓
系统校验
↓
登录成功
↓
进入对应首页
```

交互规则：手机号格式错误提示；验证码倒计时 60 秒；未勾选协议时弹窗；新手机号首次登录自动注册；医生账号未审核通过不可进入医生工作台。

**功能目的：** 为 AI 问诊、风险分级、提醒、健康记录提供基础数据。

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| pet_name | string | 是 | 宠物名称 |
| species | enum | 是 | cat / dog / other |
| breed | string | 否 | 品种 |
| gender | enum | 是 | male / female / unknown |
| neutered | enum | 否 | yes / no / unknown |
| birthday | date | 否 | 生日 |
| weight | number | 否 | 体重 kg |
| avatar | file | 否 | 宠物头像 |
| vaccine_status | enum | 否 | 未接种 / 部分接种 / 已完成 / 不清楚 |
| deworm_status | enum | 否 | 已驱虫 / 未驱虫 / 不清楚 |
| allergy_history | text | 否 | 过敏史 |
| medical_history | text | 否 | 既往病史 |
| current_medication | text | 否 | 当前用药 |
| staple_food | string | 否 | 当前主粮 |
| usual_hospital | string | 否 | 常去医院 |

操作流程：

```text
进入宠物档案页
↓
点击新增宠物
↓
填写基础信息
↓
点击保存
↓
系统创建宠物档案
↓
进入宠物详情页
↓
用户可继续补充健康信息
```

交互规则：首次使用至少创建 1 个宠物档案；AI 问诊前如无宠物档案，引导快速建档；体重只允许输入正数；用户可创建多个宠物档案；删除宠物需二次确认。

### F-003 首页健康看板

**功能目的：** 作为宠主端核心入口，展示宠物当前状态、今日待办、AI问诊入口和快捷功能。

页面结构：

```text
顶部：当前宠物切换、消息入口
健康状态卡：宠物头像、名称、今日健康状态、最近异常记录
核心入口：AI 问诊、咨询兽医、附近医院、健康记录
今日待办：喂食、用药、疫苗、驱虫、复诊提醒
科普推荐：今日科普、急救知识
```

交互规则：多宠用户可切换宠物；今日待办按时间排序；高风险未处理问诊在首页顶部强提示；无待办时展示空状态；未建档用户进入首页时弹出建档引导。

### F-004 AI 多模态问诊

**功能目的：** 通过文字、图片、语音输入，帮助用户描述宠物异常，完成 AI 多轮问诊、风险分级和行动建议。

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| pet_id | string | 是 | 当前问诊宠物 |
| symptom_text | text | 是 | 症状描述 |
| image_files | file[] | 否 | 图片，最多 6 张 |
| voice_file | file | 否 | 语音，最长 60 秒 |
| symptom_category | enum | 否 | 呕吐 / 腹泻 / 皮肤 / 泌尿 / 外伤等 |
| occurred_at | datetime | 否 | 症状开始时间 |
| extra_info | json | 否 | 多轮问诊补充字段 |

操作流程：

```text
用户点击 AI 问诊
↓
选择宠物
↓
输入文字 / 上传图片 / 语音描述
↓
点击开始分析
↓
系统创建问诊记录
↓
AI 判断症状类别
↓
系统进行多轮追问
↓
用户回答问题
↓
系统触发急症红旗判断
↓
输出风险等级和建议
↓
生成病情摘要
```

结果页结构：

```text
风险等级卡
可能相关方向
建议行动
需要立即就医的情况
病情摘要
按钮：咨询兽医 / 查看附近医院 / 创建观察提醒 / 保存到档案
```

交互规则：症状描述不能为空；图片最多 6 张，单张不超过 10MB；语音最长 60 秒；多轮追问最多 8 轮；命中急症红旗时中断普通问诊，进入高风险提示；AI 输出必须结构化。

### F-005 急症红旗识别与风险分级

| 编号 | 场景 | 触发关键词 / 条件 | 风险等级 |
|---|---|---|---|
| RF001 | 猫疑似尿闭 | 频繁进猫砂盆、尿不出、尿很少、痛叫 | emergency |
| RF002 | 呼吸困难 | 张口呼吸、喘不上气、舌头发紫 | emergency |
| RF003 | 抽搐 / 昏迷 | 抽搐、倒地、无反应、意识不清 | emergency |
| RF004 | 中毒 / 误食 | 误食药物、巧克力、清洁剂、毒物 | high / emergency |
| RF005 | 严重外伤 | 车祸、坠楼、流血不止、骨折疑似 | emergency |
| RF006 | 持续呕吐 | 多次呕吐、无法进食饮水、精神差 | high |
| RF007 | 严重腹泻 | 便血、脱水、幼宠腹泻、精神沉郁 | high |
| RF008 | 幼宠拒食 | 幼猫幼犬拒食、精神差、未完成疫苗 | high |
| RF009 | 术后异常 | 伤口裂开、持续出血、发热、精神差 | high |
| RF010 | 明显疼痛 | 哀叫、无法站立、持续蜷缩 | high |

| 等级 | 展示文案 | 系统动作 |
|---|---|---|
| 低风险 | 可短时间观察 | 创建观察提醒 |
| 中风险 | 建议线上咨询兽医 | 展示咨询兽医按钮 |
| 高风险 | 建议尽快线下就医 | 展示附近医院 |
| 急症风险 | 建议立即就医 | 强提示、医院优先 |
| 无法判断 | 信息不足 | 补充资料或转医生 |

规则：红旗规则与模型判断冲突时，以红旗规则为准。

## 7. 接口需求

### 用户接口

`POST /api/auth/send-code`：发送验证码。  
`POST /api/auth/login`：登录 / 注册。

### 宠物档案接口

`GET /api/pets`：获取宠物列表。  
`POST /api/pets`：创建宠物档案。  
`GET /api/pets/{pet_id}`：获取宠物详情。  
`PUT /api/pets/{pet_id}`：编辑宠物档案。  
`DELETE /api/pets/{pet_id}`：逻辑删除宠物档案。

### AI 问诊接口

`POST /api/consultations`：创建问诊。  
`POST /api/consultations/{id}/next-question`：获取下一轮追问。  
`POST /api/consultations/{id}/generate-result`：生成问诊结果。

返回字段：risk_level、red_flags、ai_summary、recommendation、next_actions、doctor_required、hospital_required。

### 文件上传接口

`POST /api/files/upload`：上传图片、语音、文档。

### 线上问诊订单接口

`POST /api/orders/doctor-consult`：创建医生问诊订单。  
`POST /api/payments/pay`：发起支付。  
`GET /api/orders/{order_id}`：订单详情。  
`POST /api/orders/{order_id}/refund`：退款。

### 医生工作台接口

`GET /api/doctor/consultations`：医生问诊列表。  
`GET /api/doctor/consultations/{id}`：问诊详情。  
`POST /api/doctor/consultations/{id}/reply`：医生回复。  
`POST /api/doctor/consultations/{id}/complete`：完成问诊。

### 医院接口

`GET /api/hospitals/nearby`：获取附近医院。

### 提醒接口

`POST /api/reminders`：创建提醒。  
`GET /api/reminders`：提醒列表。  
`PUT /api/reminders/{id}`：编辑提醒。  
`POST /api/reminders/{id}/complete`：完成提醒。  
`DELETE /api/reminders/{id}`：删除提醒。

## 13. 最终产品边界总结

V1.0 的产品目标不是做“AI 替代兽医”，而是做：

```text
宠物建档
↓
AI 辅助分诊
↓
急症风险识别
↓
病情摘要
↓
线上兽医承接
↓
线下医院推荐
↓
用药 / 喂食 / 复诊提醒
↓
长期健康档案沉淀
```

V1.0 必须守住四条底线：

1. AI 不确诊。
2. AI 不开处方药。
3. 高风险必须提示就医。
4. 问诊、医生、医院、用药、提醒必须形成闭环。

这版 PRD 的开发重点是把最小可用闭环做扎实，而不是一次性做完整宠物生态。
